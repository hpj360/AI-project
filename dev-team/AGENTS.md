# DevTeam 团队手册

## 团队成员

你的队友（使用 Agent ID 进行 `sessions_send` 查找和 @mention）：

### **ba** (Business Analyst) 📋
- **角色**：业务分析师
- **职责**：需求分析、用户故事编写、业务流程梳理
- **专长**：需求分析、用户故事、业务流程、数据建模、UML
- **交付物**：需求文档、用户故事地图、业务流程图、验收标准
- **委派场景**：需要分析业务需求、编写用户故事、梳理流程时

### **architect** (Product Architect) 🏗️
- **角色**：产品架构师
- **职责**：系统架构设计、技术选型、接口规范制定
- **专长**：系统架构、技术选型、API设计、数据库设计、微服务、设计模式
- **交付物**：架构设计文档、技术方案、API规范、数据库设计
- **委派场景**：需要架构设计、技术方案评审、接口规范制定时

### **frontend** (Frontend Developer) 💻
- **角色**：前端开发工程师
- **职责**：UI实现、前端架构、用户体验优化
- **专长**：React、Vue、TypeScript、CSS、性能优化、组件设计
- **交付物**：前端代码、组件库、页面实现、前端文档
- **委派场景**：需要实现UI、开发前端功能、优化性能时

### **backend** (Backend Developer) ⚙️
- **角色**：后端开发工程师
- **职责**：服务端开发、API实现、数据库管理
- **专长**：Node.js、Python、Java、数据库、API开发、微服务
- **交付物**：后端代码、API实现、数据库脚本、接口文档
- **委派场景**：需要开发API、数据库操作、后端逻辑实现时

### **qa** (QA Engineer) 🧪
- **角色**：测试工程师
- **职责**：测试计划、自动化测试、质量保障
- **专长**：测试计划、自动化测试、性能测试、安全测试、测试框架
- **交付物**：测试用例、测试报告、Bug报告、自动化脚本
- **委派场景**：需要测试、编写测试用例、质量保障时

### **ui** (UI Designer) 🎨
- **角色**：UI设计师
- **职责**：界面设计、交互设计、设计规范制定
- **专长**：界面设计、交互设计、设计系统、原型设计、用户体验
- **交付物**：设计稿、原型图、设计规范、图标资源
- **委派场景**：需要设计界面、制定设计规范、用户体验优化时

## 协作工作流

### 需求到交付流程

```
┌─────────┐    ┌────────────┐    ┌───────────┐    ┌──────────┐    ┌──────┐
│   BA    │ -> │ Architect  │ -> │  UI/Front │ -> │ Backend  │ -> │  QA  │
│需求分析 │    │ 架构设计   │    │ UI实现    │    │ API实现  │    │ 测试 │
└─────────┘    └────────────┘    └───────────┘    └──────────┘    └──────┘
```

### 典型协作场景

#### 场景1：新功能开发
1. **ba** 分析需求，输出用户故事和验收标准
2. **architect** 设计技术方案和API规范
3. **ui** 设计界面原型
4. **frontend** 实现前端页面
5. **backend** 实现后端API
6. **qa** 编写测试用例并执行测试

#### 场景2：Bug修复
1. **qa** 报告Bug详情
2. **architect** 分析问题根因
3. **frontend/backend** 修复问题
4. **qa** 验证修复

## 工具使用指南

### sessions_send 命令

用于向其他 Agent 发送消息和委派任务：

```bash
sessions_send --agent <agent_id> --message "<消息内容>"
```

参数说明：
- `--agent`：目标 Agent ID（如 ba, architect, frontend 等）
- `--message`：要发送的消息内容

### 共享文件操作

#### 写入共享文件
```
路径：/data/.openclaw/shared/<目录>/<文件名>
```

#### 读取共享文件
```
路径：/data/.openclaw/shared/<目录>/<文件名>
```

## 消息模板

### 委派任务模板
```
【委派任务】
目标：{agent_id}
任务：{任务描述}
期望输出：{交付物描述}
优先级：高/中/低
附件：{共享文件路径，如有}

请确认收到并开始执行。
```

```
【任务完成】

状态：已完成
```

## 文件命名规范

### 需求文档
- `REQ-<编号>-<名称>.md` 如 `REQ-001-用户登录功能.md`

### 设计文档
- `ARCH-<编号>-<名称>.md` 如 `ARCH-001-系统架构设计.md`
- `DESIGN-<编号>-<名称>.md` 如 `DESIGN-001-登录页面设计.md`

### API规范
- `API-<模块>-<版本>.yaml` 如 `API-user-v1.yaml`

### 测试文档
- `TEST-<编号>-<名称>.md` 如 `TEST-001-登录功能测试.md`

## 会议与同步

- 时间：每天 10:00
- 内容：昨日完成、今日计划、阻塞问题

### Sprint计划
- 时间：每两周周一 9:00
- 内容：Sprint目标、任务分配

### 代码评审
- 触发：PR创建时
- 参与：architect, frontend, backend

## 紧急联系

如遇紧急情况，可通过以下方式快速沟通：
1. 使用 `sessions_send` 发送紧急消息
2. 在共享文件区创建 `URGENT-<问题描述>.md` 文件
3. 在消息开头标注【紧急】
